# Managing Git & GitHub Entirely Through PowerShell

As a workflow challenge, I decided to manage a complete Git and GitHub lifecycle using only Windows PowerShell — no GitHub Desktop, no browser-based repo creation, and no GUI tools.

This exercise helped reinforce what actually happens beneath graphical interfaces and strengthened my command-line confidence.


## Step 1: Install the GitHub CLI

The GitHub CLI (`gh`) allows you to manage repositories directly from the terminal.

Install it using `winget`:

```powershell
PS D:\> winget install --id GitHub.cli
```

## Step 2: Create a Project Directory

Create a folder to contain your project:

```powershell
PS D:\> mkdir my_Project
```

Move into the directory:

```powershell
PS D:\> cd my_Project
```


## Step 3: Initialize Git

Instantiate a Git repository inside the project folder:

```powershell
PS D:\my_Project> git init
```

This creates a hidden `.git` directory that tracks version control metadata.


## Step 4: Create Files Directly from the Terminal

```powershell
PS D:\my_Project> ni projectFile.txt
```

Add content to the file:

```powershell
PS D:\my_Project> echo "This is my project file." > projectFile.txt
```

## Step 5: Check File Status

Git tracks changes but does not automatically commit them.

Use:

```powershell
PS D:\my_Project> git status
```

Since we created a new file, Git will report it as untracked.


## Step 6: Stage Specific Files

To prepare a file for committing, stage it:

```powershell
PS D:\my_Project> git add projectFile.txt
```

Why stage files individually?

If multiple files were modified, staging allows you to:

- Commit changes separately
- Write specific commit messages
- Maintain a clean, descriptive history

To stage all changes:

```powershell
PS D:\my_Project> git add .
```


## Step 7: Commit the Changes

Once staged, commit with a descriptive message:

```powershell
PS D:\my_Project> git commit -m "Initial commit: add projectFile.txt"
```

This creates a snapshot of the staged changes.

### Extra Tip: How to add comments to a commit message via terminal

```powershell
PS D:\my_Project> git commit -m "Initial commit: add projectFile.txt" ` -m "my comment"
``` 

## Step 8: Authenticate with GitHub via CLI

To connect your local repository to GitHub, authenticate using the GitHub CLI.

Recommended method:

```powershell
PS D:\my_Project> gh auth login
```

Follow the interactive prompts.

### Alternative: Using a Personal Access Token (PAT)

If authenticating manually:

```powershell
PS D:\my_Project> echo ghp_yourTokenHere | gh auth login --with-token
```

Notes:

- When generating a PAT on GitHub, grant appropriate scopes (e.g., `repo`).
- Name your token clearly (e.g., `powershell-cli-workflow`).
- Never expose your token publicly.


## Step 9: Rename Your Local Branch to `main`

Modern GitHub repositories use `main` instead of `master`.

Rename your local branch prior to syncing with GitHub to avoid errors:

```powershell
PS D:\my_Project> git branch -M main
```


## Step 10: Create a GitHub Repository from the Terminal

Create a remote repository directly from PowerShell:

```powershell
PS D:\my_Project> gh repo create my.Project.Repo --public --source=. --remote=origin
```

Flags explained:

- `--public` → Makes repository public
- `--source=.` → Uses current directory as source
- `--remote=origin` → Sets remote name to `origin`

---

## Step 11: Push to GitHub

Push the local repository to the remote:

```powershell
PS D:\my_Project> git push -u origin main
```

The `-u` flag sets upstream tracking, so future pushes can simply use:

```powersshell
PS D:\my_Project> git push
```
